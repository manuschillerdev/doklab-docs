Start with a compose file

You have a Docker Compose file. Nothing special — just a service, an image, maybe a build context. This is where every homelab starts.

Routing with automatic TLS

Two labels. Your service gets a public HTTPS endpoint. doklab configures Traefik as your reverse proxy, handles TLS certificate provisioning via Let's Encrypt, and routes traffic to your container. No nginx configs. No certbot cron jobs. No manual certificate renewals.

Secrets from environment variables

Passwords don't belong in git. But they need to get into containers somehow. doklab resolves secrets at deploy time from SOPS, HashiCorp Vault, AWS Secrets Manager, GCP Secret Manager, Azure Key Vault, 1Password, Doppler, and more. Works with labels or files — your compose stays compatible with local dev.

Secrets as mounted files

Some apps need certificates or config files, not environment variables. doklab mounts secrets as files with optional base64 decoding. Same provider support, different delivery method. This also works for .env files — resolve secrets into a mounted .env and your app reads them naturally.

Scale to zero

Why run containers that nobody's using? Set an idle timeout. After 15 minutes of no traffic, the container stops. The first request wakes it up — typically in under a second. Save resources. Save power. Save money.

Autoheal unhealthy containers

Have you tried turning it off and on again? Docker Compose has no native mechanism to restart unhealthy containers — it only restarts crashed ones. doklab monitors health checks and automatically restarts containers that fail them.

Simple volume backups

Volumes are precious. Losing them hurts. Nautical watches your containers and snapshots volumes on schedule. One label enables it. Backups are stored locally or pushed to remote storage via Kopia.

Custom backup commands

Sometimes you need more than a volume snapshot. Run pg_dump before backup. Export application state. Execute any pre-backup command you need. The command runs inside the container, then the volume gets snapshotted.

Backup groups

Multiple services that need consistent backups? Group them. They stop together, backup together, and restart together. No partial snapshots. No data inconsistencies between your app and database.

See what you don't see

Look at the compose file. Notice what's missing? No exposed ports. No 8080:8080 bindings cluttering your server. No plaintext passwords in environment variables. No secrets.env files committed to git. Your attack surface stays minimal. Your secrets stay secret.

Still just a compose file

Look at it. It's still Docker Compose. No new DSL to learn. No migration required. Your existing knowledge transfers directly. But now it has routing, secrets, backups, autoheal, and scale-to-zero. Platform features. Compose simplicity.

One config to rule them all

Your compose files define services. The doklab config defines how they're deployed, synced, and maintained.

Define your domain

Start with your base domain. doklab uses this for routing all your services. Pick your network mode: Tailscale for private mesh networking, or Traefik with Cloudflare for public access.

Add projects

Point doklab at a git repo or local path. Git repos are cloned, pulled, and deployed automatically. SSH, HTTPS, or local filesystem — your choice.

Auto-discover projects

Use glob patterns to discover multiple compose files automatically. Each match becomes a separate project. Think ApplicationSets from ArgoCD, but for Docker Compose.

Layer compose overlays

Different environments need different configs. Overlays merge on top of your base compose file. Perfect for production tweaks, resource limits, or environment-specific settings.

Route external services

Not everything runs in Docker. Legacy apps, VMs, other machines — they need routing too. Define external routes to proxy traffic to any HTTP endpoint. TLS termination included.

Schedule infrastructure jobs

Backups, cleanups, health checks — things that need to run on a schedule. Three job types: run spawns a container, exec runs in an existing container, local runs on the host.

Software Composition and Vulnerability Scanning included

Know exactly what's running on your machine. Every dependency. Every version. Every risk. doklab runs Trivy on schedule, generating SBOMs and CVE reports. Your homelab, audited automatically.

Your homelab, configured

One YAML file. Multiple projects, routes, jobs, and automation. No dashboards to click through. No state to manage. Just config in git, deployed on push. GitOps for homelabs. Finally.
